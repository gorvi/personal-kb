## 坑1：客户跑单（损失¥3,000）

### 踩坑经过
客户A要做一个餐饮小程序，谈好价格5000元。
- 我："我先做，做完满意了再付款。"
- 做了1周，交付了。
- 客户："功能不太对，我再想想。"
- 然后失联了。

### 原因分析
1. 没有正式合同
2. 没有定金约束
3. 交付标准不清晰

### 避坑方法

**必须做的3件事**：

1. **签电子合同**
   - 用「腾讯电子签」或「法大大」
   - 合同模板：（文末领取）

2. **分期付款**
   ```
   签约：30%定金（开始开发）
   初稿：30%（确认UI设计）
   交付：40%（上线前付清）
   ```

3. **交付标准写清楚**
   ```markdown
   【交付标准】
   □ 功能清单（附件）
   □ 通过真机测试
   □ 客户签字确认
   □ 尾款已付
   ```

## 坑2：需求蔓延（损失¥5,000）

### 踩坑经过
客户B要一个「简单的展示小程序」。
- 第1周："再加个预约功能吧"
- 第2周："再加个会员系统"
- 第3周："再加个营销工具"
- 最后做了2个月，只收了5000块。

### 原因分析
1. 需求没有书面确认
2. 没有变更流程
3. 不好意思拒绝

### 避坑方法

**需求确认单**（必须签字）：
```markdown
【项目需求确认单】

客户：_________
项目：_________
日期：_________

【功能清单】
1. □ 功能A
2. □ 功能B
3. □ 功能C

【变更规则】
- 本清单外的新增功能，视为变更
- 变更需重新报价
- 双方签字确认后生效

客户签字：_________
开发方签字：_________
```

**变更话术**：
> "这个需求在原来的清单外，属于新增功能。我评估一下工作量，明天给您报价。您确认后再做。"

## 坑3：技术选型错误（损失¥4,000）

### 踩坑经过
客户C要一个「简单的小程序」。
- 我选了原生开发（觉得性能好）
- 做了2周，发现原生开发效率太低
- 重构用uni-app，又花了1周
- 时间成本4000元

### 原因分析
1. 过度追求性能
2. 不熟悉AI辅助开发
3. 没有评估工作量

### 避坑方法

**技术选型决策树**：

```
客户预算 < 5000？
  ├─ 是 → 用uni-app（2-3天交付）
  └─ 否 → 有特殊性能要求？
      ├─ 是 → 原生开发
      └─ 否 → uni-app
```

**推荐技术栈**（2026年）：
- 框架：uni-app（Vue3）
- UI：uView Plus
- 后端：微信云开发
- AI辅助：Cursor

**原则**：不要炫技，怎么快怎么来。

## 坑4：售后拖垮（损失¥2,000）

### 踩坑经过
客户D的小程序上线了。
- 第1天："这里颜色不对，改一下"
- 第2天："那里字小了，改一下"
- 第3天："能不能再加个功能"
- ……持续1个月，改了20多次

### 原因分析
1. 没有约定修改次数
2. 没有约定售后期限
3. 没有收费机制

### 避坑方法

**合同必加条款**：
```markdown
【售后服务】
1. 免费修改：7天内，3次以内
2. 超出3次：按200元/次收费
3. 售后期限：上线后30天
4. 30天后：按500元/月维护费

【新增功能】
视为新项目，重新报价
```

**客户沟通话术**：
> "上线后7天内，小问题免费修改3次。超过3次或者新增功能，按实际工作量收费。这也是为了保证项目按时交付，望理解。"

## 坑5：定价过低（时间成本无算）

### 踩坑经过
客户E要一个「简单的商城小程序」。
- 我报价：2999元
- 实际工作量：5天
- 时薪：2999 / (5*8) = 75元/小时
- 还不如上班

### 原因分析
1. 低估了工作量
2. 没有算机会成本
3. 被市场价格吓到了

### 避坑方法

**定价公式**：
```
报价 = 开发时间 * 时薪 * 1.5（风险系数）

示例：
- 开发时间：3天 = 24小时
- 时薪：200元（程序员市场价）
- 风险系数：1.5
- 报价 = 24 * 200 * 1.5 = 7200元
```

**2026年参考定价**：
- 简单展示：2999-3999元
- 商城类：4999-7999元
- 定制功能：按1000元/天算

**原则**：宁可少接单，不做低价单。

## 坑6：不做测试就交付（损失客户信任）

### 踩坑经过
客户F的小程序上线了。
- 我没做充分测试
- 客户一用，3个明显Bug
- 客户："这么明显的Bug都没发现？"
- 信任度直线下降

### 避坑方法

**测试清单**（必须全部打勾）：
```markdown
【上线前测试清单】

□ 真机测试（至少3台手机）
□ 功能测试（所有功能点一遍）
□ 支付测试（真实支付流程）
□ 分享测试（转发朋友圈/好友）
□ 兼容性测试（不同微信版本）
□ 性能测试（加载速度、内存）

□ 客户验收（客户签字确认）
```

**原则**：宁可晚交付1天，不交有Bug的版本。

## 坑7：不签合同（法律风险）

### 踩坑经过
客户G口头约定做小程序。
- 做了2周，客户说："不做了"
- 我："那前面做的怎么办？"
- 客户："又没签合同，我又没让你做"
- 白干2周

### 避坑方法

**必须签合同**，哪怕是电子合同。

**合同关键条款**：
1. 项目范围（功能清单）
2. 付款方式（分期）
3. 交付标准
4. 违约责任
5. 知识产权归属

**推荐工具**：
- 腾讯电子签（免费）
- 法大大（专业）

## 坑8：源代码不加密（被白嫖）

### 踩坑经过
客户H说："我先看看代码质量，再付尾款"
- 我把完整代码给他了
- 客户："代码质量不行，不做了"
- 后来发现客户拿着我的代码找别人改改上线了

### 避坑方法

**分阶段交付代码**：
1. **首付款后**：给设计稿
2. **中期款后**：给测试版（部分代码）
3. **尾款后**：给完整代码

**源代码保护**：
- 核心逻辑混淆
- 关键接口加密
- 尾款前不给完整源码

## 坑9：不做知识沉淀（重复踩坑）

### 踩坑经过
每个项目都从零开始。
- 第1个项目：搭环境花了2天
- 第2个项目：又花了2天搭环境
- 第3个项目：还是……

### 避坑方法

**建立模板库**：
```
/template
  ├── /common        # 公共组件
  ├── /pages         # 页面模板
  ├── /utils         # 工具函数
  ├── /docs          # 文档
  └── README.md      # 使用说明
```

**建立文档库**：
- 常见问题FAQ
- 客户沟通话术
- 报价参考表
- 合同模板

## 坑10：单打独斗（天花板低）

### 踩坑经过
一个人做，月入1.5万就到天花板了。
- 一天只有24小时
- 一天最多做8小时
- 一个月最多做10个项目

### 避坑方法

**3个突破方向**：

1. **模板化**
   - 同行业复用代码
   - 减少重复工作
   - 效率提升3倍

2. **外包协作**
   - 设计外包（淘宝200-500元）
   - 简单功能外包（大学生兼职）
   - 你专注核心+客户沟通

3. **产品化**
   - 卖模板而不是做定制
   - 卖课程而不是做服务
   - 从卖时间到卖产品
